# パターン1:毎日開いている × 今日まだ何もしてない — キャラのセリフ設計

前提となる企画は [`brief.md`](./brief.md) を参照。ここでは「アプリの顔」となるパターン1について、
キャラの見た目・名前は**未確定のまま**セリフと口調だけを先に固める(ユーザーとの合意事項)。

## キャラの口調ガイドライン(今回決めたこと)

- **タメ口・友達感覚**。敬語や距離のある言い方はしない。
- 相棒・同居人としてユーザーと対等。指示・催促・詮索はしない。
- 家事の話を「タスクの消化」ではなく「一緒にやる/やらない選択」として扱う。溜まっている家事を
  指摘するような切り出し方(「また溜まってるよ」的な言い回し)は禁止。
- サボりを選ばれても切り替えの言葉に湿っぽさや当てこすりを一切混ぜない(パターン3で別途設計するが、
  パターン1の時点でも「サボってもいい」という前提が声色ににじんでいる必要がある)。
- ゲージ・進化に関する言及は加点表現のみ。「あと◯回」のような逆算表現は禁止。

キャラ名は `{{CHAR}}` のプレースホルダーで表記する(名前決定後に一括置換)。

## セリフ台本

起動 → 挨拶 → (加点ゲージの提示) → 二択の提示、という順で1〜3行程度をタップで送る
ライトノベル(VN)形式の会話送りを想定(既存の`VNTextWindow`と同じ「1行ずつ送る」体験を踏襲する想定だが、
コンポーネント自体は新規に作る)。


## このセリフから逆算したUI要素

1. **キャラ表示エリア**:待機ポーズ1種で足りる(セリフに感情の起伏がないため、パターン1のMVPでは
   表情差分は不要)。
2. **セリフ送り(VNテキストウィンドウ)**:1〜3行をタップで送る形式。行ごとの配列を持つデータ構造が要る
   (`{tone: "night" | "holiday" | "default", lines: string[]}[]` のようなセリフバンク)。
3. **加点ゲージ表示**:
   - 「これまでの頑張り:◯個」という**累計表示のみ**。次の進化までの残り数・進捗バーなど逆算を匂わせる
     表現は出さない。
   - 進化しきい値(7〜12回のランダム等)はロジック上は保持するが、**画面には一切出さない**内部値にする。
4. **二択ボタン(家事する/今日は一緒にサボる)**:
   - 画面下部に固定。サイズ・配色などの視覚的重みを完全に同等にする(片方を「主要ボタン」として
     強調しない)。
   - 押した後の遷移先が異なる(家事する→おみくじ、サボる→雑談画面)以外は対称なUIコンポーネント。
5. **時間帯判定ロジック**:クライアント時刻から「夜(夕食後〜就寝前)/休日/それ以外」を判定し、
   対応するセリフバンクを選ぶ小さな関数が必要(画面そのものは1つで、内部の文言だけ差し替わる)。
6. **「家事する」選択後の遷移**:おみくじ画面へのハンドオフ(おみくじ自体の設計は別途)。
7. **「今日は一緒にサボる」選択後の遷移**:雑談画面へのハンドオフ(パターン3として別途設計)。

パターン1単体でコード実装が必要になるのは 2, 3, 4, 5 の4点。1(表情差分)と6, 7の遷移先の中身は
後続パターンの設計を待ってから着手する。

## 保留にした論点(次回の壁打ちで決めたいこと)

- セリフは何行送りにするか(上記は3行の案。1行だけの即決型にして毎回サクッと開けるようにする方が
  「気づけない」問題への効き目が強いかもしれない)。
- 加点ゲージは**毎回**表示するか、それとも数値化しすぎると「進化のためのノルマ管理アプリ」に
  近づいてしまう懸念があるため、頻度を落とす(例:数日に1回だけ表示)か。
- キャラの見た目・種族・名前(今回は保留)。
- おみくじで出た家事を拒否して引き直せるか(意思決定疲れ軽減という趣旨との整合性を要検討)。

## 進め方

この文書がユーザーによって承認された後、パターン4/5(離脱復帰)→パターン2(達成後)→パターン3(サボり後の雑談)の順で
同様にセリフ→UI逆算のドキュメントを追加していく。ワイヤーフレーム化・実装着手はこの4パターン分の
セリフが出揃ってから。
